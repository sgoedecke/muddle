---
title: Cross-platform and browser compatibility
slug: cross-platform
---

I want to spend some time on cross-platform compatibility, by which I mean the question of whether your system works correctly across the variety of environments in which real users will actually try to run it. This is a topic that is easy to dismiss in the early stages of a project — "we will support modern browsers" is a sentence I have heard hundreds of times, and it is usually a way of avoiding the question rather than answering it — but the cost of getting it wrong escalates very quickly as the project gains users, and I would not wish on you the experience of trying to retrofit cross-platform support onto a system that was built without it in mind.

Let me lay out the dimensions of the problem as I see them.

There is, first, the question of browsers, if you are building anything that runs on the web. The set of browsers in active use is much larger than the set most teams test in, and the variations between them are subtle, sometimes maddening, and not always documented. Chrome, Firefox, Safari, Edge — these are the obvious four, but within each of those there are multiple versions in active use, and within each version there are platform-specific differences (Safari on macOS behaves differently from Safari on iOS in ways that matter), and within each platform-version combination there are user-configurable variations (extensions installed, accessibility settings enabled, default font sizes changed) that interact with your code in ways nobody can fully anticipate. A feature that works flawlessly in your development environment may fall apart on a five-year-old Android phone running an older WebView, and the user on that phone will not understand why your product appears to be broken; they will simply form the opinion that it is.

There is, second, the question of operating systems, if you are building anything that runs natively. The platform-specific quirks here are deeper than people realise. File path conventions differ. Line ending conventions differ. Default character encodings differ. The behaviour of fundamental system calls differs in ways that the cross-platform abstractions only partially paper over. Permissions models differ. The set of available libraries and their versions differ. A piece of code that works on a developer's Mac may fail on a customer's Windows machine in ways that the developer cannot reproduce without buying a Windows machine, which most developers will not do, with the result that the Windows-specific bug lives forever.

There is, third, the question of hardware. Screen sizes vary across more than three orders of magnitude. Input devices vary — mouse, touch, stylus, trackpad, game controller, voice — and each has different conventions about gestures, sensitivity, and feedback. Network conditions vary from gigabit ethernet to two-bar cellular. Processing power varies from a desktop workstation to a years-old budget Android phone. Memory varies. Battery life varies. Every assumption you make about the hardware on which your code will run is, somewhere, wrong, and a system that ignores the variation is a system that works well for some users and badly for others, without anyone on the team being able to predict which.

There is, fourth, the question of locales, which I have already touched on at some length in an earlier note but which is worth raising again here, because the locale of the user's device interacts with cross-platform compatibility in ways that compound the problem. Date pickers behave differently when the system locale is different from the application locale. Number inputs behave differently. Right-to-left rendering may or may not be supported by the user's browser, and may or may not interact correctly with custom CSS. The combinatorial explosion of "OS × browser × locale × device" is large enough that you cannot test every combination, and the question is therefore not "have I tested every combination" but "what are the worst plausible combinations and have I tested those?"

There is, fifth, the question of assistive technology. Screen readers behave differently across operating systems. The same ARIA attributes can produce different announcements in VoiceOver, NVDA, JAWS, and TalkBack. Code that works correctly with one screen reader can be completely opaque to another. If you have not tested your interface with the screen reader your users actually use, you do not know whether it works for them; you only know whether it works for the abstract idealised user that lives in the W3C specifications.

There is, sixth, the question of constrained environments — corporate networks that block certain hosts, regions that block certain CDNs, browsers configured to block JavaScript, browsers configured to send a Do-Not-Track header, browsers run inside a virtual machine running inside another virtual machine, browsers run by automated testing tools that you did not intend to support. Each of these introduces its own failure modes, and any system that depends on a particular network topology, a particular set of available domains, or a particular execution model is a system that will fail in some non-trivial fraction of the environments where it is actually run.

I would not raise all of this if I did not think there were a path through. The path, I think, is to pick deliberately. Decide which platforms, browsers, devices, and environments you actually intend to support. Write that decision down, share it with users, and stand behind it. Test in those environments routinely, not just at the moment of release. Be willing to gracefully degrade in environments outside your support matrix, rather than silently breaking. And when you change your support matrix, communicate the change clearly to the people affected by it, because the people you stop supporting will notice, and they will deserve at least the courtesy of a warning.

The alternative is to ship into an undefined matrix of environments and find out, after the fact, where your system happens to work, which is no way to run a serious project.
