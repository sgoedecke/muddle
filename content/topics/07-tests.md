---
title: Test coverage
slug: tests
---

I want to talk about testing, but I want to try to talk about it in a way that goes beyond the slogans, because I think the conversation around testing in our industry has become depressingly slogan-driven. People say "you need tests" and "you need coverage" and "you need to do TDD" without, in many cases, having a clear and shared understanding of what any of those phrases actually mean or what they are supposed to accomplish. So allow me to offer some thoughts that I hope will be more useful than the slogans.

The first thing I want to say is that the purpose of tests is not, primarily, to demonstrate that the code is correct. The purpose of tests is to make it safe to change the code. This is a subtle distinction but I think it is the most important one in the whole subject. A codebase with no tests can, in principle, be correct — it can be written by careful, talented engineers who think hard about every change. But a codebase with no tests cannot safely be modified by anyone other than the original author, because nobody else knows what the original author's invariants were, and any change risks violating an invariant that exists only in someone else's head. Tests are the mechanism by which invariants are made externally visible and externally enforceable. They are the substrate on which collaboration is possible.

That framing has consequences. If you accept it, then "coverage" becomes a peculiar metric, because what you actually want is not coverage in the sense of "every line was exercised by some test" but coverage in the sense of "every invariant the system relies on is enforced by some test." These are very different. A line that was exercised by a test but whose behaviour was not asserted is not actually tested; it is merely run. Most coverage tooling does not distinguish these cases. A test suite can have ninety-five percent line coverage and zero percent invariant coverage, and a great many test suites in the wild are, I suspect, closer to that end of the spectrum than anyone is comfortable admitting.

The second thing I want to say is that the marginal value of additional tests is wildly non-uniform. The first test you write for a function is enormously valuable: it proves the function does something, it documents the basic contract, it gives you a place to add regressions. The second test is valuable. The third is valuable. By the time you are writing your fifteenth test for the same function, you are almost certainly testing trivia, and you are imposing a maintenance burden on every future change to that function that may exceed the value of the tests. There is a U-shaped curve here, and it is very difficult to find the bottom of it. Too few tests and you cannot change anything safely; too many and you cannot change anything quickly. The sweet spot is real but it requires taste, and taste cannot be enforced by a coverage threshold.

The third thing I want to say is that the level at which you test matters enormously. Unit tests, integration tests, end-to-end tests, contract tests, property tests, fuzz tests, snapshot tests — these are not redundant, and they are not interchangeable. They each catch a different kind of bug and they each fail to catch a different kind of bug. A system tested only at the unit level can have every function pass its tests and the system as a whole be broken, because nothing tests the interactions between the functions. A system tested only at the end-to-end level catches integration bugs but cannot tell you which component is at fault and runs so slowly that nobody actually runs the suite during development. A serious test strategy has a deliberate distribution across these levels, with explicit reasoning about what is being caught where, and it does not just accumulate organically out of whatever each individual engineer found convenient to write.

The fourth thing I want to say, and I will be careful here because it is contentious, is that tests can be wrong, and tests can be actively harmful. A test that asserts the wrong invariant — that locks in a behaviour that is not in fact the intended behaviour — is worse than no test at all, because it gives false confidence and it actively resists correction. I have seen codebases where genuine bugs lived for years because someone had written a test for the buggy behaviour, and removing the bug required changing the test, and changing the test required arguing with whoever wrote it, and nobody had the energy. Be very, very careful about what your tests are asserting. Be especially careful about snapshot tests, which assert that the output looks like it did last time, without anybody having ever thought about whether the way it looked last time was correct.

The fifth thing I want to say is about test data. Real systems have data. Test systems usually do not, or have a tiny, artificial, sanitised version of it. Bugs that depend on the shape of real data — Unicode names that have unusual normalisation, very long strings, very small or very large numbers, rows with combinations of nullable fields that the original author did not consider — will almost never be caught by tests written against sanitised data. There is no clean solution to this, but the partial solution is to be aware of it: when a bug is found in production, the very first question to ask is "could this have been caught by a unit test on synthetic data, or did it require the shape of real data?" The latter category of bug requires a different mitigation, usually involving canary deployments, gradual rollouts, and very good observability.

I will stop here. The short version of all of this is: please do not aim for a coverage number. Aim for the property that any change to the codebase is detectable, and any regression is caught, and any invariant is documented in executable form. Those are the things that matter, and a coverage tool will not measure any of them.
