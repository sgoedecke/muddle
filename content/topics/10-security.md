title: Security

I want to talk about security, and I want to do so with the caveat that I am not a professional security researcher and that my advice should not substitute for a proper audit by someone who is. With that said, I have watched enough projects get themselves into trouble over the years that I feel I would be doing you a disservice if I did not raise at least some of the broad classes of concern that I have seen recur. Please take what is useful and ignore the rest.

The first thing I want to say is that security is not a feature; it is a property that emerges from a great many small decisions made consistently throughout the codebase. A system can be ninety-nine percent secure and still be entirely vulnerable, because the one percent that is wrong is the part the attacker will find. Defence in depth is the only realistic strategy: every layer of the system needs to assume that every other layer might be compromised, and to behave accordingly. This is, I admit, exhausting to maintain. But the alternative is to ship a system that depends, somewhere, on a single perimeter that, when it falls, exposes everything inside it, and perimeters fall remarkably often.

Let me work through some of the standard categories of concern, in roughly the order I encounter them in real codebases.

Input validation. Every piece of data that enters your system from outside should be validated against an explicit schema before it is allowed to do anything. This sounds obvious and it is, in fact, the single most commonly neglected discipline in real codebases. A surprising amount of code is written under the implicit assumption that input is well-formed, that strings have reasonable lengths, that numbers are in expected ranges, that nested objects do not contain unexpected fields. When this assumption fails — and it fails not only in the presence of malicious actors but also in the presence of confused clients, buggy upstream services, and stale cached payloads — the consequences range from amusing to catastrophic. Validate at the boundary. Reject anything that does not conform. Do not trust the client.

Authentication. The question of who is making a request is, in a serious system, the most important question you can ask about that request, and the answer needs to be reliable. The classic failure modes are well-known: session tokens that do not expire, tokens that are transmitted in URLs and end up in logs, tokens that can be forged because the signing secret was hardcoded into the client, password storage that uses anything other than a modern, well-audited password hashing function. There is no excuse for any of these in a system being built today, and yet they continue to appear with depressing regularity. Please, please use the library. Do not roll your own.

Authorization. Once you know who is making a request, the question of what they are allowed to do is the next most important question, and it is one that real codebases get wrong all the time, usually in subtle ways. A common pattern is to check authorization at the top of an API handler but not at the data access layer, with the result that any code path that bypasses the handler — a background job, an admin script, a different API endpoint that calls the same underlying function — also bypasses the check. The right place to enforce authorization is as close as possible to the data: every query should know which user it is being executed on behalf of, and should refuse to return rows that user is not allowed to see. This is harder than it sounds because most ORMs and database libraries do not make it easy, but the systems that take this seriously have dramatically fewer privilege-escalation bugs than the systems that do not.

Secrets management. Every system has secrets — API keys, database passwords, signing keys, encryption keys — and every system needs to handle them with care. The classic and embarrassing failure modes include: secrets committed to the repository, secrets stored in environment variables that are printed in logs, secrets shared across environments so that a leak in one compromises all, secrets that have no rotation procedure and have therefore been the same for several years. All of these are easy to fix in principle and astonishingly hard to fix in practice, because the right place to put a secret is in a dedicated secrets manager, and adopting a secrets manager requires changes to every part of the system that uses any secret, which is usually most of it.

Dependencies. Modern software is built on top of mountains of third-party code, much of it written by volunteers in their spare time, much of it audited by no one, much of it depending on yet more third-party code in turn. Every dependency you add is a piece of code you are running with the privileges of your own application, and a vulnerability in any of those dependencies is a vulnerability in your system. There is no way to eliminate this risk entirely, but you can manage it: keep dependencies up to date, pin versions to specific releases, run a software composition analysis tool that alerts you to known vulnerabilities in your dependency graph, and be deeply sceptical about adding a new dependency for a small benefit. The systems with the smallest blast radii are usually the ones with the smallest dependency trees.

Logging and observability, paradoxically, are also security concerns. A log file that contains user passwords, session tokens, or personal data is a security incident waiting to happen, because logs accumulate, logs get shipped to third-party aggregators, logs get included in backups, logs get exfiltrated when a system is compromised. The discipline of scrubbing sensitive data from logs at the point of generation, rather than relying on the log aggregator to do it later, is the only one that actually works.

I will stop here, but the short version is: assume you will be attacked, assume your defences will be partial, and design accordingly. Hoping that no one will notice is not a strategy; it is the absence of one.
