---
title: Code style and readability
slug: code-style
---

I want to talk about code style, but I want to try to do so in a way that does not devolve into the tedious tabs-versus-spaces, brace-on-same-line-or-next-line arguments that have consumed so much of our industry's time and produced so little of value. Those debates are debates about typography. They are not debates about anything that actually matters. What I want to talk about, instead, is readability, which is a substantive topic on which a great deal hinges and which gets, in my experience, much less serious attention than it deserves.

The thesis I want to advance is that code is read far more often than it is written, and that the cost of writing code is therefore dramatically smaller than the cost of reading it, summed over the lifetime of the project. Any choice that makes the code marginally more readable is, almost certainly, worth a substantial amount of additional writing effort. The corollary is that any choice that makes the code marginally less readable, even if it makes the code shorter or faster to write, is almost certainly a bad trade. I have seen this principle violated more times than I can count, usually by engineers who are very proud of their cleverness and who do not yet have the experience of being the person who, two years later, has to read the clever thing and figure out what it does.

So let me try to enumerate some of the dimensions on which I think readability operates, in case any of them are useful to you.

The first is naming. Names matter enormously, more than almost any other aspect of code style, because the name is the interface by which the reader engages with the concept. A function named `process` tells the reader nothing; a function named `convertCSVToDatabase` tells the reader something useful. A variable named `x` is meaningless; a variable named `attemptCount` is informative. The cost of typing a longer name is paid once, by the author, in fractions of a second. The cost of a confusing name is paid forever, by every reader, in the time required to figure out what the thing is. The trade is so wildly asymmetric that I find it astonishing how much code in the world is poorly named. I would urge you to set a high bar here: rename things until the names are right, even if it is a hassle, because the names you choose will outlive you.

The second is the length of functions. There is no magic number, but I will offer an opinion: any function that does not fit on a single screen of a normally-sized editor is probably doing too many things, and any function that contains more than two or three levels of nested control flow is probably trying to express something that would be clearer as several functions. Long functions are not always wrong, but they are almost always a smell, and when I see one I usually find, on inspection, that it would be clearer if broken up. Short functions are, by contrast, almost always right, even when the breakup feels slightly artificial, because each function becomes a unit of thought that the reader can hold in their head independently.

The third dimension is the choice of abstractions. This is one of the hardest things to do well, and I do not have a crisp prescription, but I have an observation: the wrong abstraction is much worse than no abstraction at all. Inlined code can always be refactored later. Wrong abstractions accumulate accidental commitments that become very hard to undo. There is a well-known piece of programming wisdom that says "duplication is far cheaper than the wrong abstraction," and I have come to believe it. When you find yourself reaching for an abstraction, it is worth asking: do I have enough examples of this pattern to know what the abstraction should look like? If you have one example, you have no idea. If you have two examples, you have a guess. If you have three examples that genuinely share structure, you might have a real abstraction. Below that threshold, please just duplicate and wait.

The fourth dimension is what I will call locality. A function whose behaviour depends only on its inputs is dramatically easier to read than a function whose behaviour depends on global state, hidden context, or implicit side effects. Every time you reach for a global variable, a mutable singleton, or an out-of-band side effect, you are imposing a cost on every reader of every piece of code that touches the thing. Sometimes this cost is necessary. Often it is not, and the function would have been simpler and clearer with the dependency passed in explicitly. The "ports and adapters" or "functional core, imperative shell" patterns are, at their heart, attempts to maximise this kind of locality, and I think they are usually worth the structural overhead.

The fifth dimension is comments, and I have an unfashionable opinion: most comments are bad. The good ones explain why the code does what it does, especially when the why is non-obvious. The bad ones explain what the code does, which the code already explains better. The really bad ones are wrong, because the code has been changed and the comment has not, and now the reader has two conflicting sources of truth. Be sparing with comments. Use them to capture intent, history, and context that cannot be expressed in code itself, and leave the rest to the code.

I will stop here, but I want to add a meta-point: every team eventually develops a house style, and the house style is more important than any individual stylistic choice. A codebase that is internally consistent in a slightly suboptimal style is dramatically easier to navigate than a codebase that is locally optimal in places but inconsistent overall. So please decide, write it down, enforce it with tooling, and then stop arguing about it.
