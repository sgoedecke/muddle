---
title: Edge cases and null handling
slug: edge-cases
---

I want to spend some time on what I consider to be the most under-appreciated source of long-term bugs in any non-trivial system, which is the handling of edge cases — and in particular, the handling of values that are not, strictly speaking, errors but are also not, strictly speaking, normal. Nulls. Empty strings. Empty arrays. Zero. Negative zero. The empty set. The unset optional. The user who has not yet completed onboarding. The order that exists but has no line items. The friend list that is empty because the user just signed up, not because something went wrong. These values are everywhere in real systems, and they are, in my experience, the single most common source of the bugs that get filed in production at three in the morning and that turn out, on inspection, to be one missing if-statement two function calls deep.

The problem is that the type system, in most languages, does not protect you from these cases as well as you might hope. A function that takes a list might receive an empty list. A function that takes a string might receive an empty string. A function that takes a user might receive a user who has no email address yet, even though the schema says every user has an email address, because some user was migrated from an old system and the migration script had a bug. The type system says "this is a list" or "this is a string" or "this is a user," and the type system is, technically, correct. But the implicit assumption your code is making — that the list has at least one element, that the string has at least one character, that the user has all the fields you expect — is not encoded in the type, and the compiler cannot help you when that assumption breaks.

I have come, over the years, to a few rules of thumb that I would like to share, in case any of them are useful.

The first is to be ruthless about distinguishing between the absence of a value and the presence of a default. These are not the same thing, and treating them as if they were is the source of an enormous amount of subtle bugs. A user who has not set a notification preference is different from a user who has explicitly set their notification preference to "off." A field that is missing from a payload is different from a field that is present in the payload with the value `null`. Your system should preserve this distinction at every layer — in the database, in the application code, in the API responses — because the moment you collapse the distinction you lose the ability to ever recover it, and you will eventually need to recover it, because someone in product will ask "how many users explicitly opted out, versus how many users just never made a choice?"

The second rule of thumb is to be paranoid about empty collections. An enormous amount of code is written under the implicit assumption that the collection being processed is non-empty. The average over an empty list is undefined; the max is undefined; the first element does not exist; the "most recent" is meaningless. These cases should be handled deliberately, at the point where the collection is processed, and the handling should be a conscious decision: "if the list is empty, return null and let the caller decide," or "if the list is empty, return zero and document the convention," or "if the list is empty, this is a programmer error and we should crash loudly." Any of these is acceptable. What is not acceptable is to let the question be answered implicitly by whatever the language's default behaviour happens to be, because the language's default behaviour is rarely what the user actually needs.

The third rule of thumb is to think hard about identity in the presence of optional fields. If two records are equal whenever their non-null fields are equal, then a record with a null field is equal to many other records, and you will get equality semantics that confuse everyone. If two records are equal only when all their fields, including nulls, are equal, then nulls become a kind of value, and you will get equality semantics that confuse everyone in a different way. There is no right answer; there is only the answer you choose deliberately and document loudly, versus the answer you stumble into accidentally and then have to defend in code review for the next five years.

The fourth rule of thumb is to remember that "not yet" is a different state from "not." A user who has not yet uploaded a profile picture is, in some important sense, in a pending state, not in an absent state. A payment that has not yet been processed is not the same as a payment that has been processed and declined. The richer your model of these intermediate states, the better your system will handle them; the more you try to collapse them into a single nullable field, the more your application logic will twist itself into knots distinguishing them downstream.

I will stop here, but I want to add one observation that may be useful. The single most reliable way I have found to surface latent edge-case bugs in a system is to ask, for every field in every data structure, the question: "what does it mean if this is missing or empty?" The answer should be explicit, written down, and checked in code. If the answer is "it cannot be missing or empty," then there should be a constraint or assertion that proves it cannot be missing or empty. If the answer is "it can be missing or empty, and here is what we do in that case," then there should be code that implements that, also checked in. The systems I have seen that took this discipline seriously had dramatically fewer production bugs than the systems that did not. The cost of the discipline is meaningful but not enormous; the benefit, over years, is hard to overstate.
