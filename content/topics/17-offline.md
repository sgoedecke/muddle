---
title: Offline and network resilience
slug: offline
---

I would like to talk about offline support and network resilience, because these are two areas where my impression is that the gap between what users implicitly expect and what most systems actually deliver is very wide, and getting wider. Users have, over the last decade, become accustomed to applications that "just work" even when the network is bad — that hold on to their input, that retry their actions, that synchronise their state, that recover gracefully from interruptions. Native mobile applications, in particular, have set a high bar for this kind of resilience, and web applications that fall short of the bar increasingly feel, to users, like broken software.

Let me lay out what I think the relevant concerns are.

The first concern is the basic question of what happens when the network goes away in the middle of a user action. The naive default — the action fails, the user gets an error message, the application returns to its previous state, and the user has to start over — is, in many contexts, not acceptable. A user filling out a long form does not want to be told that their submission failed and that the form has been cleared. A user uploading a photo does not want to be told that the upload failed and that they should try again later. A user composing a message does not want to discover that the message was never sent because the network blipped at the wrong moment. The standard mitigations — persisting form state locally, queueing actions for retry, using service workers to mediate between the application and the network — are not difficult to implement individually, but they require thinking about offline as a first-class concern from the start, not as a feature to be added later.

The second concern is the question of conflict resolution. The moment you allow users to perform actions while offline, you have to grapple with the possibility that, when they come back online, their version of reality may have diverged from the server's version of reality. Someone else may have modified the same record. The same user may have modified the same record on another device. The server may have changed the schema or the validation rules in the meantime. There is no general-purpose answer to "how do I reconcile divergent histories"; there are only domain-specific answers, and the answer for any particular system requires real thought about what the user actually wants in each case. Last-write-wins is usually wrong, because it silently destroys the loser's work. Manual conflict resolution is usually unacceptable, because users have no idea what to do. The right answer is usually some combination of operational transforms, CRDTs, deliberate merge strategies, and human-friendly conflict UI, all carefully composed.

The third concern is the question of cache invalidation, which is, as the old joke has it, one of the hard problems in computer science. Any system that caches data on the client to enable offline use has to grapple with the question of when that cache is stale and what to do about it. Stale data presented as fresh is, in some sense, worse than no data at all, because the user makes decisions based on it and discovers, too late, that the decisions were based on a stale snapshot. The mitigations are complex — etags, versioning, push notifications for invalidation, periodic background refresh, careful UX that surfaces the data's freshness — and they have to compose correctly across multiple data types with different freshness requirements. Getting this right is among the harder problems in client-side engineering, and most systems do not.

The fourth concern is the question of partial connectivity, which is in some ways harder than full disconnection. A user on a slow or unreliable network may have a connection that intermittently works, that returns errors at unexpected times, that times out under load. Code that is written under the assumption that "either the network works or it does not" will not handle this case correctly. The actions that are most likely to fail are the ones that are most important to the user, because they take the longest and consume the most bandwidth. Retries help, but unconstrained retries cause the retry storms I have written about elsewhere. Backoff and jitter help, but they delay the user's perception of progress. Showing meaningful feedback during long, partially-failing operations helps, but most user interfaces do not do this well.

The fifth concern is the question of what the application looks like before it has any data at all. The first time a user opens the application — or the first time after the cache has been cleared — the application has to load from the network, and during that time it has nothing to show. The default behaviour of most web applications is to show a blank page, then a partially-rendered page, then a fully-rendered page, in a sequence of layout shifts that is both disorienting and slow. The better answer involves server-rendered initial markup, skeleton placeholders for content that is still loading, and progressive enhancement that makes the page usable before the JavaScript has finished executing. This is a substantial engineering investment, and most teams do not make it.

The sixth concern, and the one I find most consistently neglected, is the question of how the user knows what is happening. A well-designed offline-capable application makes its state visible at all times: this is what is on the server, this is what is on your device, this is what is queued for sync, this is what has failed and needs your attention. Most applications do not surface any of this. They present the user with a unified view that conceals all the underlying state, and when something goes wrong the user has no idea what to do, because they have no idea what state the system is in. Surfacing state requires UI work, taxonomy work, and writing work, and it is often the first thing cut when deadlines approach. I would urge you not to cut it. The user's ability to trust the system depends entirely on their ability to see what it is doing.

I will stop here, but the short version is: the network will fail, and how you handle that failure is one of the most user-visible decisions you will make.
