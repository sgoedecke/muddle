---
title: Documentation
slug: documentation
---

I would like to spend some time on documentation, which is, in my experience, one of the most consistently undervalued aspects of building software. There is a kind of running joke in our industry that documentation is what you write when you have finished the real work, and that the real work is the code. I think this is exactly backwards. Documentation is part of the work. A piece of software that nobody can use, because they cannot figure out how to use it, is not a piece of software that has any value; it is a piece of code that happens to exist. The transformation from "code that exists" to "software that someone benefits from" runs entirely through documentation, and skimping on it is skimping on the deliverable.

That said, the word "documentation" covers an enormous range of things, and most discussions of it collapse into incoherence because the participants are talking past each other about different kinds. Let me try to disentangle a few categories.

There is reference documentation, which is the kind of documentation that answers the question "what does this thing do?" for someone who already knows roughly what they are looking for. Reference documentation is, for the most part, mechanically generatable from the code itself, and the best reference documentation is generated this way, because it then stays in sync with the code automatically. The mistake I see most often in reference documentation is over-explanation: pages and pages of prose describing what the function does, when a clear signature, a short summary, and three lines of example code would have communicated everything any reader needed. Less is usually more.

There is conceptual documentation, which is the kind of documentation that answers the question "how does this thing fit together?" for someone who is trying to build a mental model of the system. Conceptual documentation cannot be generated; it has to be written, by a human, with care, in prose. The mistake I see most often here is the opposite: too little. Codebases of enormous complexity ship with a README that says "to install, run npm install; to run tests, run npm test" and then expects every contributor to reverse-engineer the architecture from the source. This is a profound failure of communication, and it imposes a cost on every new contributor and every old contributor returning to a part of the system they have not touched in a year. Writing a good architecture overview takes a day. Not having one costs a day for every new person who joins, forever.

There is tutorial documentation, which is the kind of documentation that walks a complete beginner through their first successful interaction with the system. Tutorials are different from reference documentation and different from conceptual documentation, because their purpose is not to teach you everything but to give you a single, small, complete success experience that builds enough confidence and context to make the rest navigable. Most software ships without tutorials, or ships with tutorials that have not been touched in years and that no longer match the current version, and the result is that new users bounce off the system before they ever experience the value it could provide. A working tutorial that takes a beginner from "I have never used this" to "I have made this thing do the thing I wanted" is, in my view, the single most valuable piece of documentation a project can have, and the single most consistently neglected.

There is operational documentation, which is the kind of documentation that explains how the system behaves in production, how it fails, how it is recovered, and what the on-call engineer is supposed to do when an alert fires at three in the morning. This category of documentation is almost universally bad, in part because it is almost universally written under emotional duress, immediately after an incident, when the people writing it are too exhausted to think clearly. The best operational documentation is written before the incident, as part of the normal work of building the system, and reviewed and updated regularly. The worst — but most common — pattern is to have no operational documentation at all, to be discovered by the on-call engineer in the middle of their first real outage.

There is, finally, decision documentation: the record of why the system is the way it is. This is the kind of documentation that prevents future-you from undoing the careful work of past-you because you no longer remember why past-you did it. It is also the kind of documentation that lets new contributors disagree productively with the existing design, rather than either accepting it as inscrutable holy writ or tearing it down without understanding what they are losing. Architecture decision records, design docs, post-mortems, RFC threads — these all serve this purpose, and they all pay enormous dividends over the lifetime of a system, but they all require the discipline to actually write them down at the moment the decision is made, rather than reconstructing them later from memory, which is to say, never.

I should also raise the question of what documentation is for, philosophically. There are at least three audiences: users (who need to know how to use it), contributors (who need to know how to change it), and operators (who need to know how to run it). The needs of these audiences are different, and a single document that tries to serve all three usually serves none of them well. Be deliberate about whom each piece of documentation is for, and write each piece for that audience.

I will stop here, but I want to leave you with one observation that I think is undervalued in the discourse: documentation is a form of empathy. It is the act of imagining the reader who does not yet know what you know and giving them what they need to catch up. Writing it well requires the discipline of remembering what it was like not to know, and that discipline is harder to maintain than it looks, especially as you become more expert. But it is, in my view, the most underrated engineering skill there is.
