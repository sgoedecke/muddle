---
title: Maintainability and tech debt
slug: maintainability
---

I want to talk for a while about maintainability, and specifically about technical debt, which is one of those metaphors that has been used so widely and so loosely that it has begun to lose its meaning. People use "tech debt" to refer to all manner of things that are not, in fact, debt in any useful sense — slow tests, outdated dependencies, code they personally dislike, decisions they were not in the room for. I would like to try to recover some of the original force of the metaphor and use it to talk about something that I think is real and important and chronically under-managed.

The original idea of technical debt, as I understand it, was that you can sometimes ship faster by taking a shortcut, with the understanding that you will later pay interest on that shortcut in the form of additional work to clean it up. The shortcut itself is not necessarily a bad thing — sometimes shipping faster is worth the interest payment — but it becomes a bad thing if it is taken without intent, without a plan for repayment, and without a clear accounting of what was deferred. The systems I have seen run aground on technical debt did not run aground because they had any one piece of debt that was insurmountable. They ran aground because hundreds of small pieces of debt had accumulated, none of them ever quite urgent enough to fix, and the cumulative weight of them had made every change slow and risky and demoralising.

So I want to suggest a few practices that I think help to keep the situation manageable.

The first is to make the debt visible. A piece of debt that exists only in someone's head is debt that nobody is paying down, because nobody is keeping track of it. Whenever you take a shortcut — whenever you skip a test, whenever you copy-paste a block of code instead of refactoring it, whenever you defer a migration to a later sprint — write it down somewhere durable. A TODO comment is the minimum acceptable practice. A ticket in your issue tracker, with enough context that a stranger could understand what is owed and why, is better. The point is not to commit to fixing the debt; the point is to create a record so that, when the question "what is wrong with this part of the codebase?" comes up, the answer is not "nobody knows."

The second is to distinguish between the kinds of debt. There is debt that you took on deliberately, with full understanding of the trade, because shipping was urgent. There is debt that you took on accidentally, because you did not understand what you were doing at the time. There is debt that was not debt when you took it on but that has become debt because the environment around it has changed — a library has been deprecated, a platform has been retired, a customer has emerged with requirements you did not anticipate. These three categories require different responses. Deliberate debt should be on a clear schedule for repayment, because you committed to repaying it. Accidental debt should be repaired when discovered, as part of learning the lesson that produced it. Environmental debt should be tracked but treated as a fact of life, because the environment will keep changing whether you like it or not.

The third practice is to budget for debt repayment explicitly. The pattern I see in most teams is that all of the planned engineering time goes to new features, and debt repayment happens only when something has caught fire badly enough that it cannot be ignored. This is exactly the wrong order. Debt accrues continuously; it should be paid down continuously, with some fraction of every sprint or every quarter explicitly allocated to it. The right fraction is not zero, and it is also not one hundred percent; it is some number in between that depends on the maturity of the system, the rate of new feature development, and the team's tolerance for risk. But it should be a deliberate number, defended in planning meetings, and not a residual that gets squeezed out whenever a deadline approaches.

The fourth practice, and this one is contentious, is to be willing to throw things away. There is a strong cultural pressure in our industry to treat existing code as a sunk cost that must be preserved, modified, refactored, but never deleted. This is wrong. Sometimes the right answer is to delete the code and write a replacement, and the replacement is dramatically smaller and cleaner because you have learned what the code actually needed to do, which you did not know when you originally wrote it. The cost of writing the replacement is real but often smaller than the accumulated cost of years of careful refactoring around the original mistake. Be willing to make this trade when the evidence supports it, and do not let "we have invested a lot in this code" become a reason to invest more in it forever.

The fifth practice is about people, not code. Maintainability is not a property of the code alone; it is a joint property of the code and the people who maintain it. A piece of code that was perfectly maintainable when its author was on the team becomes much less maintainable when the author leaves, taking with them all the context that lived only in their head. Every piece of code in your system should be understandable by someone other than the person who wrote it, and the only way to verify this is to have other people actually read it and try to change it. Code review serves this purpose, but only if reviewers actually engage with the code rather than rubber-stamping it. Pair programming serves this purpose. Rotating who works on which subsystem serves this purpose. Insisting on it as a value, and noticing when it lapses, serves this purpose.

I will stop here, but the short version is: keep an honest accounting of what you owe, pay it down deliberately, and do not let the codebase become something only one person understands.
