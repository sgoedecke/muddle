Our SCA scanner flagged `fast-parse@2.1.0` for a regex denial of service. A crafted input causes catastrophic backtracking and pins a CPU core. We use it in exactly one user-facing place.

Short term: pin to `2.0.7`, which predates the vulnerable regex. Long term: the iterative parser rewrite in #819 removes the dependency entirely. The same scanner rule should gate the bundle-analyzer discussion in #847 and be wired into the reproducible build in #948.
