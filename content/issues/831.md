Migration `0043_split_orders_table` has no working `down` path. We tried to roll back a bad deploy and it left the table half-split — no way to tell which rows moved. We restored from backup and lost ~7 minutes of writes.

Every irreversible migration needs a tested rollback before it ships. This is the same "irreversible action on fallible input" hazard as the non-idempotent handler in #869. Can we add a CI gate that refuses migrations whose `down` isn't exercised? That gate depends on the reproducible-build work in #948.
