Migration `0043_split_orders_table` has no working `down` path. We tried to roll back a bad deploy last night and it left `orders` in a half-split state — some rows in the new table, some in the old, no way to tell which. We restored from backup but lost ~7 minutes of writes.

Every irreversible migration needs a tested rollback before it ships. This is exactly the "irreversible action on fallible input" class of problem that #869 is also an instance of (double-charging is irreversible too).
