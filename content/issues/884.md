Critical: when `tenant_id` is `null`, the row-level authorization predicate `tenant_id = :current` evaluates to `null` (not `false`), and our query builder treats a `null` filter as "no filter", so the query returns rows across all tenants. Any code path that can produce a `null` tenant — a bad migration, a degraded fallback, a missing header — leaks cross-tenant data.

We've already seen this fire during two unrelated incidents: the half-split migration in #831 and the degraded ingest path in #823. This is the textbook "the type system says it's fine but null isn't false" hazard.
