title: "Empty/null value silently bypasses the validation guard"
labels: [bug, edge-cases, security, priority-critical]
state: open
author: security-wg
opened: Nov 9, 2024
---

When a key field is `null`, our guard predicate `field == :expected` evaluates to `null` (not `false`), and the query builder treats a `null` filter as "no filter" — so the check is silently skipped and we return data we shouldn't. Any path that can produce a `null`/empty value (a bad migration, a degraded fallback, a missing field) slips through.

Textbook "the type says it's fine but null isn't false". We've seen it fire via the half-split migration in #831 and the degraded path in #823. The fix is to make null/empty a hard error at the boundary — but every entry point must be audited or one miss undoes it all.
