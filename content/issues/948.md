There's a race between cache invalidation and read repair. If a read-repair writes back a value it fetched *before* an invalidation, it repopulates the cache with the stale value *after* the invalidation cleared it. Result: we serve stale rows indefinitely until the next write. The window is small but under load it happens regularly.

This is the read-side twin of the write-side clobber in #812 — same "operation ordered against a concurrent operation" bug. The correct invalidation ordering here is also a prerequisite for safely warming the cache in #877; warm the wrong way and you bake in the stale value.
